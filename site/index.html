Bollard

A data-flow firewall
for AI agents.

Bollard tags where every piece of an agent's data came from, then gates each outbound call — and routes each model call — on that provenance. Enforced at the network boundary, so even a shelled-out curl can't escape.

The problem

The lethal trifecta

An agent falls the moment it holds all three at once: access to private data, exposure to untrusted content, and the ability to make outbound calls. A single injected web page or document can then steer it into leaking your secrets. It isn't a model bug — it's a data-flow problem, and nobody constrains the flow.

[A] vs [H]: the identical request is allowed before the agent touches the trifecta and denied after. [F]/[G]: the exfil is blocked over an open sink and a trusted one. [J]: enforcement is structural — there is no route out except the boundary.

How it works

Tag · propagate · gate · enforce

1 · Tag

Every tool output is stamped with its provenance and sensitivity as it enters the agent — trusted-config, user, untrusted-web, private-file.

2 · Propagate

That taint follows the data through the agent loop. Conservative and deny-by-default: it never produces a false allow.

3 · Gate

Each outbound action is decided on the taint — including the model call, which is just another egress: sensitive context is pinned to local inference.

4 · Enforce

At the OS/network boundary, not the SDK. The agent runs where there is no route out except the boundary, so nothing can skip it.

          ┌──────────────── cage (no route out) ─────────────────┐
  tools ─▶ │  bollard-mcp ─tag─▶ agent loop ─outbound calls─▶      │─▶ bollard-proxy ─▶ internet
          │      │ (provenance)                                    │       │ allow · deny · DLP
          └──────┼──────────────────────────────────────────────-─┘       │
                 └────────────────▶ bollard-broker ◀─────────────────────-─┘
                            taint store + policy decision engine
                  model calls ─▶ bollard-infer ─▶ local (on-prem) | cloud, by sensitivity

What it enforces

Four guarantees

  • Provenance-gated egress. An outbound call is allowed on the origin of the data it carries, not just its destination.
  • The privacy router. A model call made while a secret is in context is kept on a local backend that physically can't reach the internet; benign calls still use cloud.
  • Content-aware DLP. Where the payload is visible, a request carrying a registered secret is blocked — even to a trusted, allowlisted host.
  • Unbypassable by construction. The cage has no route out except the boundary, so a subprocess, a raw socket, or a shelled-out curl can't escape.
